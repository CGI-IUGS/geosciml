Introduction
COMMENT: "Edit please. Vast majority of the content that you'll need to provide for this profile, will go into the introduction."
The GEA Standard - OWS Profiles (hereafter referred to as the profile) specifies a constrained set of Web Services (WS) and Web Services interoperability standards to enable the heterogeneous computing systems present within Queensland Government to interoperate thus allowing the integration of government processes across agencies. The conformance constraints presented in this document build upon the set of Web Services technologies, standards and specifications (WS-*) and existing work from Web Services Interoperability Organization (WS-I ) in refining those technologies specifically to foster maximal levels of understanding and interoperability.
Scope
COMMENT: "Below is a sample section defining scope. Please modify for the profile you are working on"
The scope of the profile are all OGC based Web Services exposed by Queensland Government agencies, outside their boundary and consumed by another agencies. The profile does not cover OGC Web Services used solely within one agency. The profile does not cover non-GIS, generic SOAP and WSDL based Web Services as these are covered in GEA Standard - Web Services Interoperability.
Document structure and conventions
Notational conventions

The keywords " MUST", " MUST NOT", " REQUIRED", " SHALL", " SHOULD NOT", " SHOULD", " SHOULD NOT", " RECOMMENDED", " MAY", and " OPTIONAL" in this document are to be interpreted as described in RFC2119. It is essential that the reader of the Standard fully understands RFC2119 before attempting to implement Web Services conformant to the Standard. In particular, it is very important that the keywords SHOULD or RECOMMENDED are not misinterpreted as “optional” and that the subtle difference between keywords RECOMMENDED and MUST is fully understood. RFC2119 is a three page document.

Conformance to the standard is defined by adherence to the set of requirements defined for a specific target, within the scope of the standard. This section explains these terms and describes how conformance is defined and used.
Conformance Levels
COMMENT: "This is a standard section that will be automagically inserted. You don't need to do anything here "
This Standard defines three conformance levels to describe Web Services implementations’ conformance compliance to the Standard:
  • NON CONFORMANT - An implementation is said to be NON CONFORMANT if it fails to satisfy one or more of the MUST or REQUIRED level requirements for the artifacts it exposes.
  • CONDITIONALLY CONFORMANT – An implementation that satisfies all the MUST level requirements but not all the SHOULD or REQUIRED level requirements is said to be CONDITIONALLY CONFORMANT.
  • UNCONDITIONALLY CONFORMANT – An implementation that satisfies all the MUST NOT or REQUIRED level and all the SHOULD or RECOMMENDED level requirements is said to be UNCONDITIONALLY CONFORMANT.
Conformance requirements
COMMENT: "This is a standard section that will be automagically inserted. You don't need to do anything here "
Requirements state the criteria for conformance to the Standard. They occasionally refer to an existing specification or profile of standards (such as WS-I) and embody refinements, interpretations and clarifications to it in order to improve interoperability. All requirements in the standard are considered normative unless otherwise clearly defined as being non-normative, and are testable where possible. When requirements in the standard and its referenced specifications contradict each other, the standard requirements take precedence for purposes of conformance. Requirement levels, using RFC2119 terminology (e.g., MUST, SHOULD, MAY) indicate the nature of the requirement and its impact on conformance. Each requirement is individually identified (e.g., GEAXXXX) for convenience. For example: GEAXXXX WIDGETs SHOULD be round in shape. This requirement is identified by GEAXXXX, applies to the conformance target WIDGET, and places a conditional requirement upon widgets; i.e., although this requirement must be met to maintain conformance in majority of cases, situations may exist where there may be valid reasons for it not being met (which are explained in the requirement itself, or in its accompanying text). Each requirement statement contains one or more non-contradicting requirement level constructs (e.g., " MUST", “ REQUIRED”) and one or more conformance target keywords (e.g., "MESSAGE" or “DESCRIPTION”). Additional text may be included to illuminate a requirement or group of requirements (e.g., rationale and examples); however, prose surrounding requirement statements must not be considered in determining conformance. Definitions of terms in the standard are considered authoritative for the purposes of determining conformance.
